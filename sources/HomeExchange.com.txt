Use Case 3: BO Roles Redesign - Security & User Research

Context

The Back Office had 3 role types (team, admin, translator) that didn't match actual job functions. Security audit revealed over-permissioning: people with "edit" rights also had "impersonation" capabilities, creating unnecessary risk.

Daniel's Systematic Approach

Phase 1: Current State Mapping

    Audited all existing BO users (~50 people)
    Documented what permissions they theoretically had vs. actually needed
    Identified risk: Customer service reps had impersonation rights they never used

Phase 2: User Research & Needs Analysis

Created interview guide and met with each team:

Happiness Team:

    What they do: View member info, check exchange status, validate documents
    What they need: Read access + specific actions (cancel exchange, trigger emails)
    What they DON'T need: Edit member profiles, impersonate users, financial actions

Sales Collection Team:

    What they do: Evaluate applications, manage eligibility, communicate with prospects
    What they need: Collection-specific data + application status changes
    What they DON'T need: Access to regular member exchanges, GP actions

Finance Team:

    What they do: Refunds, deposits, GP adjustments, coupon management
    What they need: Financial actions only
    What they DON'T need: Member data editing, exchange management

Data Team:

    What they do: Run queries, export data, check tracking
    What they need: Read-only access to everything
    What they DON'T need: Any write permissions

Freelancers/Temporary Staff:

    Current problem: Given permanent "team" role, access never revoked
    What they need: Time-limited, scope-limited access

Phase 3: New Role Architecture Design

Proposed 5 new roles based on job function:

    Customer Service (Read-Heavy): View + limited actions
    Collection Specialist: Collection-specific full access
    Finance Operator: Financial actions only
    Data Analyst: Read-only everywhere
    Admin: Full access (Product/Tech only)

Plus temporal access system for freelancers.

Phase 4: Technical Coordination

    Worked with Platform team on Auth0 implementation
    Created migration plan: how to move 50 users from old to new roles
    Built rollback plan if issues discovered

Skills Demonstrated

    Security Thinking: Identified principle of least privilege violations
    User Research: Conducted needs analysis across 5 different user types
    System Design: Created role architecture that scales with company growth
    Risk Management: Built rollback plans and phased rollout strategy
    Technical Collaboration: Translated business needs into Auth0 requirements
    Change Management: Prepared communication for users about permission changes

Impact

    Reduced security exposure by removing unnecessary permissions
    Created scalable role system for future hiring
    Improved audit compliance with clear permission logic
    Enabled temporary access management for contractors
